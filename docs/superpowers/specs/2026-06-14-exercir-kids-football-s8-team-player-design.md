# Exercir kids-football MVP — Slice 8 (team view + player modal) design

> **Status:** approved (brainstorming) 2026-06-14. Slice 8 of the Club Grass kids-football MVP — the **team view + player modal**: a roster (last-5 attendance dots, present %, latest-note preview), and a player modal with coach-notes CRUD, per-event attendance toggles, and a **development** view (focus-area practice counts + recent effect checks aggregated from completed runs the player attended). This is the **LAST slice of the 8-slice ladder** and the retention payoff — and the slice where the long-deferred **`subjectSensitivity` trigger fires**, engaging the ADR-187/188/189 developmental-ethics / consent gates.
> **Repo:** `domains/exercir` (CROSS-LAYER: a small `@de-braighter/substrate-contracts` publish for the new `PackManifest.subjectSensitivity` field). **Substrate:** consumes the `@de-braighter/substrate-{contracts,runtime}` line (currently pinned `^2.1.1`; bumped to the new publish) — auth/tenant/RLS + the persisted audit (slice 6) + the **already-wired consent foundation** (ADR-184).
> **Related:** `2026-06-11-exercir-kids-football-mvp-design.md` (§8 row 8, §4 Member.notes + Event.attendance/run shapes, §5 the expected-effect seam cashed into the development view) · `2026-06-14-exercir-kids-football-s7-run-session-design.md` (the `event.run.ratings`/`run.log` + `event.attendance` data this aggregates) · `2026-06-14-exercir-kids-football-s6-calendar-design.md` (the event vertical + `updateEvent`) · handoff `domains/exercir/docs/design/exercir-mvp-handoff/README.md` screen 8 + prototype `exercir/proto/team.jsx` (the authoritative roster + player-modal + development-aggregation logic) · ADRs **187** (ontology-integrity invariants), **188** (developmental-ethics charter — the `subjectSensitivity` binding criterion + C1–C8), **189** (product AI-safety ladder), **184** (consent foundation), **176** (kernel minimality / derive-on-read).

## 1. Goal

A coach / assistant-coach / team-manager reviews each player's **attendance** + **development** for the active team. Two surfaces:

- **Roster table** — avatar (team-color tint), name, born, last-5 attendance dots (present = accent, absent `#C25441`, excused `#E8A93C`), present % (≥75 green else red), latest-note preview. Staff chips above. **Alphabetical** (never ranked by performance — ADR-188 C1). Scoped to the active team (the team switcher re-scopes it).
- **Player modal (560px)** — header avatar + stats line ("3 of 4 sessions (75%) · 1 excused"); **Coach notes** (input + Add → dated rows, delete); **Attendance** (per past event: 5 status chips, toggleable); **Development — practised while present**: focus-area count chips ("Passing ×3") + recent effect checks (date, drill, ✓ On track / – Not yet) aggregated from completed runs the player attended.

This completes the coach loop's retention payoff and **fires the `subjectSensitivity` trigger** — the first slice to surface per-player developmental data about minors. The kernel is still used only as the tenant/auth/RLS + persisted-audit + **consent** substrate; **no kernel domain concepts, no inference, no Bayesian twin** — the development view is **fact-aggregation** (counts + recorded coach yes/no checks), derived on read.

## 2. Decisions (2026-06-14 brainstorming)

1. **`subjectSensitivity` fires — YES — as a typed `PackManifest` field (cross-layer publish).** The development view surfaces per-player *developmental* data about minors (focus-area practice counts + per-drill effect-check history per child), which is the doc-pinned trigger deferred across slices 5/6/7. Even though it is **fact-aggregation, not Bayesian inference** (no posterior, no twin, no `RunManifest`, no kernel primitive), the trigger fires. ADR-188 D2 names `PackManifest.subjectSensitivity` as the binding criterion, but the shipped `@de-braighter/substrate-contracts@2.x` never implemented the field — slice 8 is the **first consumer**, so it lands the field. **Founder decision: add it as a typed field** (`subjectSensitivity?: 'developmental-minor' | 'vulnerable-adult'`) to substrate-contracts `PackManifest`, publish, bump exercir's pin, and set `'developmental-minor'` on the kids-football manifest. Documented as a **charter/CI-governance field** — read by `charter-checker` and validated by the verifier wave, *not* by a kernel-runtime subsystem (the ADR-176 inclusion-test nuance: a ratified charter pin (ADR-188) demanded this exact field; the first consumer lands it as a governance-typed slot).
2. **Consent C7 — wire a real consent guard (ADR-188 C7 path, founder decision).** Declare one consent purpose `kids-football.view-player-development` on the manifest and gate the development surface with the **already-shipped** `@RequiresConsent` decorator. NOT the demo-mode exemption. Realization in §3/§4 below.
3. **Permissions 14 → 16 — two narrow perms.** Add `noteWrite` + `attendanceWrite`, granted to **coach + assistantCoach + teamManager** (the handoff "team-manager edits notes + attendance" — without over-granting `memberWrite`/`eventWrite`, which would let team-manager create members / schedule / run). Narrow RBAC endpoints delegate to the **existing** `updateMember` / `updateEvent` use-cases (no new write use-cases).
4. **Development aggregation is derive-on-read (ADR-176), computed server-side behind the consent guard.** Because `@RequiresConsent` is route-param-based (`subjectFrom` = an HTTP route param holding the subject id), the development aggregation moves to a server endpoint `GET /members/:memberId/development` (the subject = the player route param). It is **derived from events, never persisted** — ADR-176 holds; it is a server *query*, not a stored rollup. (This reverses the "derive-on-read in the UI / no endpoint" default that slices 2/5/6/7 used — the cost of choosing the real consent guard.)
5. **Consent is scoped to the development surface only.** The roster's attendance facts (dots, present %) + coach notes are **operational** (gated by `memberRead`/`noteWrite`/`attendanceWrite`, not consent). The development aggregation (what the child is developing at) is the developmental-state-of-a-minor surface ADR-188 protects, so it carries the consent guard. Honest caveat documented for `charter-checker`: `/events` still exposes raw `run.log`/`ratings` to event-readers (needed for run-session + calendar), so C7 here is a **presence guard on the developmental presentation**, not a data-siloing guarantee — the UI renders development **only** from the gated endpoint, never client-aggregates it.
6. **No new table.** Member.notes (slice 1 JSONB), event.attendance + event.run (slice 6/7 JSONB), drill.focus (slice 4) already persist. Consent receipts stay **in-memory** (the app already wires `InMemoryConsentReceiptRepository` unconditionally); a **demo consent seed** grants the purpose for the seeded players at bootstrap (else the empty repo fails the guard closed → 403). Prisma consent persistence (`PrismaConsentReceiptRepository` + `core.consent_receipt`) is a **doc-pinned forward seam** for `demoMode=false`.
7. **Humane framing (ADR-188 C1/C2/C4/C5-tone, ADR-189).** The roster is **alphabetical, never ranked** (C1 hard prohibition + ADR-189). The development section carries a caption "A record of practice while present — not a skill rating." (C2/INV-8 spirit — no bare developmental label, even though no inferred State is surfaced). Copy stays non-deficit ("✓ On track / – Not yet", "practised while present") (C4 / C5-tone review-checklist). No recommendation, no AI → ADR-189 risky rungs untouched, C8/INV-3 trivially satisfied.

## 3. Data model — no new table

The slice reads already-persisted shapes (all from `pack-kids-football-contracts`):

```
member.notes      : [{ id, date(ISO), text }]                              // slice 1 JSONB
event.attendance  : { [memberId]: 'present'|'absent'|'sick'|'holiday'|'school' }  // slice 6/7 JSONB
event.run         : { startedAt, drillStartedAt, idx, completed, finishedAt?,
                      log: [{ drillId, plannedMin, actualSec, skipped }],
                      ratings: { [drillId]: 'yes'|'no' } }                 // slice 7 JSONB
drill.focus       : string                                                 // slice 4
```

Derived, never stored: `attStats` (recorded/present/excused/absent/pct), the last-5 dots, the development `{ focusCounts, checks }`. Consent receipts: `core.consent_receipt` shape (kernel; in-memory adapter here) — `{ tenantPackId, subjectId, packKey, purposeId, grantedByActorId, grantedAt, withdrawnAt }`.

### Aggregation rules (from prototype `team.jsx`, authoritative)
- `attStats(memberId, events)`: over events with recorded attendance that include this member — `present` = count 'present'; `excused` = count in `{sick,holiday,school}`; `absent` = recorded − present − excused; `pct = round(present/recorded*100)` or `null` if no recorded sessions.
- Last-5 dots: events with recorded attendance, sorted desc by date, take 5, reverse (chronological).
- Development (the **consent-gated** aggregation): over events where `run.completed && attendance[playerId] === 'present'` — `focusCounts[drill.focus] += 1` for each non-skipped `run.log` entry; `checks.push({ date, drillName, rating })` for each `run.ratings` entry; `checks` sorted desc by date, shown first 6.

Bump exercir's pin to the new contracts (and runtime if the line moved). No runtime/SQL change in substrate for this slice (the consent guard + `subjectSensitivity` need no new runtime).

### 4.2 Contracts (`pack-kids-football-contracts`)
- New `kf-development.ts` (+ spec): pure helpers — `attStats(memberId, events)`, `lastFiveAttendance(memberId, events)`, `aggregateDevelopment(playerId, events, drills)` → `{ focusCounts: Record<string,number>, checks: { date, drillName, rating }[] }`. Clock-injected where a "today/now" cutoff is needed (the attendance-history `e.date <= today` filter takes an injected `nowMs`/`todayIso`) — **never `new Date()` inside**; never hardcoded dates in tests (the SSE-PT24H lesson).
- New permission ids `noteWrite`, `attendanceWrite` in `KF_PERMISSIONS` (roles.ts); plus the consent purpose id constant `KF_CONSENT_PURPOSES.viewPlayerDevelopment = 'kids-football.view-player-development'`.
- A `PlayerDevelopmentSchema` (the dev endpoint's response shape: `{ focusCounts, checks }`) + type, exported from the barrel (public-API additive).
- No entity-schema change (notes/attendance/run already typed).

### 4.3 API (`pack-kids-football` + `pack-kids-football-api`)
- **Manifest** (`pack-kids-football/src/manifest/pack-manifest.ts`): set `subjectSensitivity: 'developmental-minor'`; add `consentPurposes: [{ id: KF_CONSENT_PURPOSES.viewPlayerDevelopment, displayName: 'View a player's development record', legalBasis: 'revFADP Art. 6' }]`; add the two permissions; grant `noteWrite` + `attendanceWrite` to coach + assistantCoach + teamManager. (`Object.values(P)` already gives clubAdmin both.) Update `pack-manifest.spec.ts` (16-perm assertions + subjectSensitivity + consentPurposes + role-closure).
- **New `GetPlayerDevelopmentUseCase`** (`pack-kids-football/src/in-ports` + `application`): `getDevelopment(memberId): Result<PlayerDevelopment, …>` — loads the member's events (via the event repository / list use-case) + drills, calls the contracts `aggregateDevelopment` helper, returns the derived shape. Pure derivation, no persistence (ADR-176). `member-not-found` failure if the member doesn't exist for the tenant.
- **New endpoints** in the API controllers (no new write use-cases — delegate to existing):
  - `GET /kids-football/members/:memberId/development` → `GetPlayerDevelopmentUseCase`, gated `@RequiresPermission(memberRead)` **+** `@RequiresConsent({ purposeId: KF_CONSENT_PURPOSES.viewPlayerDevelopment, subjectFrom: 'memberId' })`.
  - `PATCH /kids-football/members/:id/notes` `{ notes }` → `updateMember(id, { notes })`, gated `@RequiresPermission(noteWrite)`.
  - `PATCH /kids-football/events/:id/attendance` `{ attendance }` → `updateEvent(id, { attendance })`, gated `@RequiresPermission(attendanceWrite)` (rides the slice-6 `event.updated` audit).
- **Demo consent seed:** a bootstrap (mirroring the demo-drill/club seed) grants `view-player-development` for every seeded **Player-role** member of the stub clubs via the singleton `CONSENT_RECEIPT_REPOSITORY` (`grant({ tenantPackId, subjectId: member.id, packKey: 'kids-football', purposeId, grantedByActorId: <stub admin id> })`). Idempotent / swallow `ActiveConsentConflictError`. So the demo development view renders; the deny path is proven by e2e (a member without a receipt → 403).
- **Optional boot hardening:** a `ConsentBootstrap` (`OnApplicationBootstrap`) calling `collectDecoratorConsentPurposeIds` + `ConsentEngine.validateDecoratorReferences` (catches purpose-id typos at boot; mirrors the PolicyEngine bootstrap). Include if low-cost.
- **e2e:** the development endpoint — coach with a consented player → 200 `{focusCounts, checks}`; coach with an **un-consented** player → 403 (the consent gate); a role without `memberRead` → 403. Notes PATCH — coach/teamManager → 200; a role without `noteWrite` (e.g. facilities) → 403. Attendance PATCH — coach/teamManager → 200 merged; facilities → 403.

### 4.4 UI (`pack-kids-football-ui`)

## 7. Risks / notes

- **Empty consent repo fails closed (the central operational risk).** `InMemoryConsentReceiptRepository` starts empty → `@RequiresConsent` 403s every development read until the **demo consent seed** grants receipts. The seed MUST run at bootstrap for the stub clubs (and any onboarding-created club's players, if development is exercised there). Verify in the browser run-through that the development section renders (not a perpetual 403).
- **Consent-as-presence-guard caveat.** `/events` exposes the same raw run data to event-readers, so the consent gate protects the developmental *presentation*, not the raw data. Documented for `charter-checker`; the UI must render development **only** from the gated endpoint (never client-aggregate it) to keep the gate meaningful.
- **Cross-layer publish ordering.** Verify the substrate main line + the latest published contracts version before bumping (the "verify origin/main before bumping" lesson). The `subjectSensitivity` field is additive — api-extractor will flag the public-API addition; run `api:update` on the substrate side.
- **TenantScopedClient fan-out (budgeted):** the development READ goes through the existing event/member repos — **no new TenantScopedClient delegate** is expected (no new repository), so the fan-out-to-sibling-specs trap should NOT bite. If a delegate is added, it fans to stub-delegates + every sibling Prisma contract spec (a missed one = a TS2741 only `nx build` catches).
- **CSS budget** (battle-tested): roster + modal are big — decompose + trim under 8kB; verify via the prod build.
- **Charter-checker engagement.** The PR engages `charter-checker` + `exercir-charter-checker`; ask them to specifically bless: the `subjectSensitivity` typed-field declaration (governance-typed slot), the C1 no-ranking roster, the C7 consent scoping (development-only + the presence-guard caveat), and that the development aggregation is derived-not-stored (ADR-176).
- **No new permission for the development READ.** It reuses `memberRead` (coach/ac/teamManager hold it) + the consent guard. Only the WRITES (notes/attendance) get the two new perms.
